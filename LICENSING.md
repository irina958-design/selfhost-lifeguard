# Key administration

How Pro keys are issued, identified, and revoked. Operational notes for the
maintainer; users only need [`ACTION.md`](ACTION.md).

## What identifies whom

There is no user database. Lemon Squeezy is the single source of truth, and
each layer of identity is derived from it:

| Layer | Identifier | Where it lives |
|---|---|---|
| Customer | order id and email | Lemon Squeezy order, created at checkout |
| Entitlement | license key | one per order, generated by Lemon Squeezy |
| Installation | `lifeguard-<12 hex>` | instance created by the first Pro run |

The installation name is the SHA-256 of the repository slug on a GitHub runner,
or of the hostname otherwise, truncated to 12 hex characters. It is stable for
that installation, opaque to us, and reproducible by the operator — which is
what makes support possible without collecting anything identifying.

## Lifecycle

1. **Issue.** Checkout generates the key and emails it. Nothing to do by hand.
2. **Activate.** The first Pro run posts `activate` with the installation name.
   Lemon Squeezy records an instance against the key and returns its id, which
   the runner stores in `~/.lifeguard/instances.json`.
3. **Validate.** Every later run posts `validate` with that instance id. No new
   instance is created, so runs are unlimited and installations are not.
4. **Enforce.** The variant's activation limit is the number of installations
   one key covers. Beyond it, `activate` refuses and the command exits `3` with
   the merchant's message.
5. **Revoke.** Disable the key in the Lemon Squeezy dashboard. The next run
   fails closed; nothing needs to be published or redeployed.

## Pilot keys

Pilot keys are free but must stay distinguishable from paid ones, so they are a
separate **variant** of the same product — *Lifeguard Pro — Pilot* — issued
through a 100% discount code rather than by hand. That gives, at no extra work:

- pilots and customers separated by variant in the dashboard;
- a real order per pilot, so the key can be revoked or extended individually;
- self-service delivery, so no key ever travels through a public thread.

Set the pilot variant's activation limit to 1 and its subscription length to
the pilot window. Pilot evidence stays in
[issue #5](https://github.com/irina958-design/selfhost-lifeguard/issues/5);
the installation name is the safe thing for an operator to quote there.

## What can be tracked

From the Lemon Squeezy licenses view alone: key status, owning customer and
order, activation usage against the limit, and every instance with its name and
creation date. That answers how many installations a customer runs, when each
first activated, and which keys are dormant — without any telemetry from us.

What is deliberately not tracked: run counts, command outcomes, Immich or
database versions, hostnames, paths, and anything else about the installation.
Only the key and the opaque installation name leave the runner.

## Support

A blocked run prints `installation: lifeguard-<12 hex>`. Ask for that line, find
the matching instance in the dashboard, and the key, order, and customer follow
from it. If the operator cannot reach the run output, the same value can be
recomputed from the repository slug or hostname.
